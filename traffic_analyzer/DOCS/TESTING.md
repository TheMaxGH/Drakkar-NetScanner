# План тестирования DRAKKAR NetScanner

## 1. Цель тестирования

Цель тестирования — подтвердить, что DRAKKAR NetScanner корректно выполняет основные функции: запускается, определяет сетевые интерфейсы, захватывает пакеты, применяет BPF-фильтры, отображает данные в таблице, обновляет графики, показывает детали пакета и экспортирует результат в PCAP.

Дополнительно проверяются устойчивость интерфейса, поведение при ошибках, потребление памяти и корректность обработки пользовательских сценариев.

## 2. Объекты тестирования

| Объект | Что проверяется |
|---|---|
| Запуск приложения | Открытие главного окна, проверка зависимостей |
| Проверка Npcap | Предупреждение при отсутствии драйвера |
| Список интерфейсов | Читаемые названия адаптеров, скрытие служебных интерфейсов |
| Захват пакетов | Поступление пакетов в таблицу |
| BPF-фильтр | Отбор пакетов до обработки |
| Парсер | Корректное определение IP, портов, протоколов, длины |
| Таблица | Производительность и корректность отображения |
| Дерево деталей | Отображение уровней и полей выбранного пакета |
| График интенсивности | Подсчет пакетов в секунду |
| Диаграмма протоколов | Распределение TCP/UDP/DNS/ARP/ICMP и других протоколов |
| Экспорт PCAP | Создание файла с сырыми пакетами |

## 7. Тесты визуализации

### 7.1 График интенсивности

Проверяется, что график показывает количество пакетов в секунду, не уходит в отрицательные секунды и расширяется вправо при длительном захвате.

```text
Ожидаемый вид:

Пакеты/с
5 |             *
4 |         *   *       *
3 |     *   *   *   *   *
2 | *   *   *   *   *   *
1 | *   *   *   *   *   *
0 +----------------------------> секунды
    0   10  20  30  40  50  60
```

Критерии прохождения:

1. Ось X начинается с 0.
2. Отрицательные секунды не отображаются.
3. Ось Y начинается с 0.
4. При поступлении пакетов появляются точки или линия.
5. После очистки график возвращается к пустому состоянию.

### 7.2 Диаграмма протоколов

Проверяется, что диаграмма изменяется при появлении разных типов пакетов.

```text
Пример ожидаемого распределения:

TCP      | ######################## 60%
UDP      | ########                 20%
DNS      | ####                     10%
ARP      | ##                       5%
ICMP     | ##                       5%
```

Критерии прохождения:

1. В легенде отображаются названия протоколов.
2. Проценты меняются при поступлении новых пакетов.
3. Не появляются непонятные названия вроде `Dot3`, если пакет можно классифицировать как LLC/Ethernet.

## 9. Тест памяти

Цель — убедиться, что длительный захват не приводит к неконтролируемому росту ОЗУ.

Порядок:

1. Запустить приложение от имени администратора.
2. Выбрать активный интерфейс.
3. Установить фильтр `tcp or udp`.
4. Нажать «Старт».
5. Открыть диспетчер задач.
6. Наблюдать за процессом 30-60 минут.
7. Зафиксировать начальное, среднее и максимальное потребление памяти.

Таблица фиксации:

| Время | Пакетов | ОЗУ | ЦП | Комментарий |
|---|---:|---:|---:|---|
| 0 минут | 0 | ___ МБ | ___ % | Старт |
| 10 минут | ___ | ___ МБ | ___ % | Нормальная работа |
| 30 минут | ___ | ___ МБ | ___ % | Проверка устойчивости |
| 60 минут | ___ | ___ МБ | ___ % | Длительный захват |

Критерий прохождения: память может расти до заполнения буфера, но не должна расти бесконечно с одинаковой скоростью.

## 11. Проверка отказоустойчивости

| Ошибка | Ожидаемое поведение |
|---|---|
| Нет Npcap | Показать окно с ссылкой на загрузку |
| Нет прав администратора | Показать ошибку, не завершать приложение |
| Некорректный BPF | Показать ошибку Scapy/libpcap |
| Нет пакетов | Приложение остается открытым, графики пустые |
| Экспорт без пакетов | Пользователь получает информационное сообщение |

## 12. Проверка безопасности

| Риск | Проверка | Ожидаемый результат |
|---|---|---|
| Утечка трафика | Проверить сетевые соединения приложения | Приложение не отправляет пакеты во внешние сервисы |
| Случайная передача PCAP | Проверить сценарий экспорта | Файл создается только после выбора пользователя |
| Бинарные данные в payload | Захватить разные пакеты | Интерфейс не ломается, длинные значения ограничены |
| Ошибка драйвера | Запуск без Npcap | Появляется понятное предупреждение |

## 13. Итоговые критерии приемки

Проект считается прошедшим тестирование, если:

1. Приложение запускается из виртуального окружения.
2. Интерфейсы отображаются понятными именами.
3. Захват работает на активном адаптере.
4. Фильтры `tcp`, `udp`, `icmp`, `port 53` применяются корректно.
5. Таблица обновляется без зависания.
6. Дерево деталей показывает уровни выбранного пакета.
7. График интенсивности не показывает отрицательное время.
8. Диаграмма протоколов обновляется при поступлении пакетов.
9. Экспорт PCAP создает файл.
10. Ошибки обрабатываются понятными сообщениями.
