# Архитектура DRAKKAR NetScanner

## 1. Назначение документа

Данный документ описывает архитектуру дипломного проекта DRAKKAR NetScanner: состав программных модулей, причины выбора технологического стека, схему движения сетевого пакета от сетевого интерфейса до визуализации, структуру данных, потоковую модель и ключевые проектные решения.

Документ предназначен для пояснительной записки к дипломному проекту и может использоваться при защите для ответа на вопросы о том, почему система построена именно таким образом.

## 2. Цель разработки

Цель проекта — разработать настольную систему визуализации и интерактивного анализа сетевого трафика, которая позволяет пользователю:

1. Выбрать сетевой интерфейс.
2. Запустить захват пакетов в реальном времени.
3. Применить BPF-фильтр до начала захвата.
4. Просматривать пакеты в таблице.
5. Анализировать структуру выбранного пакета по уровням.
6. Наблюдать интенсивность трафика в динамике.
7. Оценивать распределение протоколов.
8. Экспортировать результат в PCAP для дальнейшего анализа.

Основной акцент сделан не на замене Wireshark, а на создании понятной, расширяемой и демонстрационной системы, которую можно объяснить как инженерный проект: от драйвера захвата до модели таблицы и графиков.

## 3. Общая архитектурная идея

Приложение построено по принципу разделения ответственности:

| Слой | Назначение | Почему выделен отдельно |
|---|---|---|
| Захват пакетов | Получение сырых пакетов с интерфейса | Захват может блокировать выполнение, поэтому он вынесен в отдельный поток |
| Разбор пакетов | Преобразование Scapy-пакета в удобную структуру | GUI не должен зависеть от внутреннего API Scapy |
| Представление данных | Таблица, дерево, графики | Интерфейс обновляется только из главного Qt-потока |
| Документация | Описание установки, тестирования и архитектуры | Для дипломного проекта важно не только работающее ПО, но и объяснимая инженерная база |

## 6. Схема движения пакета

```text
1. Сетевая карта
       |
       v
2. Npcap/libpcap
       |
       v
3. Scapy sniff()
       |
       v
4. PacketCaptureThread
       |
       +-- хранение сырого пакета для PCAP
       |
       v
5. parser.разобрать_пакет()
       |
       v
6. PacketRecord
       |
       v
7. Qt-сигнал packet_captured
       |
       v
8. Главный GUI-поток
       |
       +-- таблица пакетов
       +-- дерево деталей
       +-- график пакетов/с
       +-- диаграмма протоколов
```

## 8. Причины выбора Python 3.10+

Python выбран по следующим причинам:

| Причина | Обоснование |
|---|---|
| Быстрая разработка | Для дипломного проекта важно реализовать полный рабочий прототип с GUI, захватом и документацией |
| Богатая экосистема | Существуют зрелые библиотеки Scapy, PyQt6 и PyQtGraph |
| Читаемость кода | Код проще объяснять на защите, чем низкоуровневую реализацию на C/C++ |
| Кроссплатформенность | Один код может запускаться на Windows, Linux и macOS при наличии драйвера захвата |
| Достаточная производительность | Захват и первичный разбор учебного трафика укладываются в возможности Python |

## 9. Причины выбора Scapy

Scapy используется как библиотека захвата и разбора пакетов.

| Критерий | Почему Scapy подходит |
|---|---|
| Захват в реальном времени | Функция `sniff()` позволяет принимать пакеты с выбранного интерфейса |
| BPF-фильтры | Фильтр применяется до передачи пакета в Python-код, что снижает нагрузку |
| Разбор протоколов | Scapy уже знает основные уровни: Ethernet, ARP, IP, IPv6, TCP, UDP, ICMP, DNS |
| PCAP-экспорт | `wrpcap()` позволяет сохранить захваченные пакеты в стандартный формат |
| Обучаемость | Пакет можно исследовать как объект Python, что удобно для демонстрации |

Альтернативой был PyShark, но для данного проекта Scapy предпочтительнее, потому что он проще встраивается в Python-приложение и не требует обязательного запуска внешнего tshark-процесса.

## 10. Причины выбора PyQt6

PyQt6 выбран как GUI-фреймворк, потому что:

| Причина | Пояснение |
|---|---|
| Нативный интерфейс | Окно выглядит как полноценное настольное приложение |
| Сигналы и слоты | Удобная и безопасная передача данных между потоками |
| `QThread` | Захват пакетов можно вынести из GUI-потока |
| Модель-представление | `QAbstractTableModel` эффективнее для больших таблиц, чем ручное создание тысяч виджетов |
| QSS-стили | Можно сделать современную темную тему без отдельного UI-фреймворка |

## 11. Причины выбора PyQtGraph

PyQtGraph выбран для графика интенсивности трафика:

| Причина | Пояснение |
|---|---|
| Производительность | PyQtGraph хорошо подходит для потокового обновления графиков |
| Интеграция с Qt | Виджет напрямую встраивается в PyQt6-интерфейс |
| Простота | Для графика пакетов в секунду не нужен тяжелый стек визуализации |
| Интерактивность | Пользователь может масштабировать и изучать график |

Matplotlib также подходит для статических графиков, но для живого обновления в интерфейсе PyQtGraph обычно отзывчивее.

## 12. Потоковая модель

Главный принцип: GUI не должен захватывать пакеты самостоятельно. Если вызвать `sniff()` в главном потоке, окно перестанет отвечать на действия пользователя.

В проекте используется такая модель:

```text
GUI-поток Qt
    |
    +-- кнопки, таблица, графики, диалоги
    |
    +-- запускает PacketCaptureThread

PacketCaptureThread
    |
    +-- вызывает Scapy sniff()
    +-- получает пакет
    +-- разбирает пакет
    +-- отправляет PacketRecord через pyqtSignal
```

Почему это важно:

1. Пользователь может нажать «Стоп» даже при интенсивном трафике.
2. Таблица не блокируется во время захвата.
3. Qt-виджеты изменяются только из главного потока, что снижает риск аварийного завершения.
4. Архитектура легко расширяется: можно добавить отдельные потоки для тяжелой аналитики.

## 13. Структура `PacketRecord`

`PacketRecord` — это промежуточная структура между Scapy и GUI. Она нужна, чтобы интерфейс работал с простыми строками и числами, а не с тяжелыми объектами пакетов.

| Поле | Тип | Назначение |
|---|---|---|
| `номер` | `int` | Порядковый номер пакета в текущем сеансе |
| `время` | `str` | Время получения пакета в читаемом формате |
| `метка_секунды` | `int` | Unix-время для группировки по секундам |
| `источник` | `str` | IP или MAC отправителя |
| `назначение` | `str` | IP или MAC получателя |
| `протокол` | `str` | Определенный протокол верхнего уровня |
| `длина` | `int` | Размер пакета в байтах |
| `информация` | `str` | Краткое описание для таблицы |
| `слои` | `list[dict]` | Данные для дерева деталей |

## 14. Производительность таблицы

Для отображения пакетов выбран `QAbstractTableModel`, а не `QTableWidget`.

| Подход | Проблема/преимущество |
|---|---|
| `QTableWidget` | Создает объект ячейки для каждого значения, при 10 000+ строк становится тяжелее |
| `QAbstractTableModel` | Хранит данные в списке и отдает значения по запросу представления |

Именно поэтому таблица может оставаться отзывчивой при большом количестве пакетов.

## 15. Ограничение памяти

Сырые пакеты сохраняются во внутреннем буфере для экспорта PCAP. Буфер ограничен по количеству пакетов.

```text
Новые пакеты добавляются в конец:

[старые пакеты ................................ новые пакеты]

Если лимит превышен:

[самые старые удалены][актуальный буфер для экспорта]
```

Причина такого решения: без ограничения памяти длительный захват может привести к постоянному росту потребления ОЗУ.

## 16. Работа с интерфейсами Windows

На Windows Scapy и Npcap часто используют технические имена вида:

```text
\Device\NPF_{E5C748A9-A519-4C41-8767-A1ACD27B6A79}
```

Для пользователя такие имена непонятны. Поэтому приложение получает из Scapy расширенное описание адаптера и показывает в списке понятную подпись:

```text
Realtek Gaming GbE Family Controller (192.168.1.178)
```

При этом внутри для захвата сохраняется техническое имя `\Device\NPF_...`, потому что именно его ожидает Npcap.

## 17. Обработка отсутствия Npcap

На Windows для захвата пакетов требуется Npcap. Приложение проверяет наличие драйвера при запуске и при попытке начать захват.

Если Npcap не найден, пользователь видит диалог с объяснением и ссылкой:

```text
https://npcap.com/#download
```

Такое решение выбрано потому, что ошибка отсутствующего драйвера иначе выглядит непонятно: пользователь может увидеть пустой список интерфейсов или системную ошибку Scapy.

## 18. Безопасность данных

Проект учитывает следующие риски:

| Риск | Мера |
|---|---|
| Захват конфиденциального трафика | Экспорт PCAP выполняется только по явному действию пользователя |
| Утечка данных во внешние сервисы | Приложение не отправляет пакеты в сеть |
| Поврежденный или бинарный payload | Значения полей ограничиваются по длине перед выводом в интерфейс |
| Нехватка прав | Пользователь получает понятное сообщение об ошибке |

## 19. Возможности расширения

Архитектура позволяет добавить:

1. Загрузку PCAP-файлов.
2. Поиск по таблице.
3. Цветовые правила для подозрительных пакетов.
4. Статистику по IP-адресам.
5. Детектирование простых аномалий: всплеск DNS, порт-сканирование, большое число SYN-пакетов.
6. Экспорт статистики в CSV или JSON.
7. Сохранение пользовательских фильтров.

## 20. Итог

Архитектура DRAKKAR NetScanner построена вокруг трех ключевых требований: отзывчивость интерфейса, понятность анализа и расширяемость. Захват вынесен в отдельный поток, разбор отделен от GUI, таблица реализована через модель данных, а визуализация строится на легких Qt-виджетах. Такой подход подходит для дипломного проекта, потому что демонстрирует не только работу приложения, но и осознанные инженерные решения.
